# Отчёт по проекту: «Приложение для рецептов»

## 0. Заявление темы проекта

Тема проекта: **приложение для рецептов**.

Цель проекта — разработать клиентское Java-приложение, которое работает с реляционной базой данных через JDBC и позволяет хранить рецепты, категории, ингредиенты, шаги приготовления, отзывы пользователей и избранные рецепты.

Примерный перечень таблиц:

- `app_user` — пользователи приложения;
- `category` — категории рецептов;
- `recipe` — рецепты;
- `ingredient` — ингредиенты;
- `recipe_ingredient` — состав рецепта;
- `recipe_step` — шаги приготовления;
- `review` — отзывы и оценки;
- `favorite` — избранные рецепты пользователей.

## 1. Описание предметной области

Предметная область проекта — хранение и использование кулинарных рецептов. Пользователь приложения должен иметь возможность просматривать список рецептов, открывать подробное описание рецепта, искать рецепт по названию, добавлять новые рецепты, оставлять отзывы и добавлять рецепты в избранное.

Основные сущности предметной области:

1. Пользователь — человек, который добавляет рецепты, пишет отзывы и сохраняет рецепты в избранное.
2. Категория — группа рецептов, например «Салаты», «Супы», «Горячие блюда», «Десерты», «Гарниры».
3. Рецепт — основная сущность приложения. Рецепт имеет название, описание, время приготовления, количество порций и сложность.
4. Ингредиент — продукт, который используется в рецепте.
5. Состав рецепта — связь между рецептом и ингредиентом с указанием количества и единицы измерения.
6. Шаг приготовления — отдельная инструкция в составе рецепта.
7. Отзыв — оценка и комментарий пользователя к рецепту.
8. Избранное — список рецептов, сохранённых пользователем.

Правила и ограничения предметной области:

- один рецепт относится только к одной категории;
- один пользователь может добавить много рецептов;
- один рецепт может включать много ингредиентов;
- один ингредиент может использоваться во многих рецептах;
- связь между рецептом и ингредиентом является связью многие-ко-многим и разрешается через таблицу `recipe_ingredient`;
- рецепт должен иметь положительное время приготовления;
- количество порций должно быть положительным;
- оценка в отзыве может быть только от 1 до 5;
- один пользователь может оставить только один отзыв на один рецепт;
- один пользователь не может добавить один и тот же рецепт в избранное два раза;
- email пользователя должен быть уникальным;
- название категории должно быть уникальным;
- название ингредиента должно быть уникальным.

## 2. Логическая модель базы данных

Логическая модель включает следующие сущности и связи.

### Сущность `Пользователь`

Атрибуты:

- идентификатор пользователя;
- имя пользователя;
- email;
- пароль в виде хеша;
- дата регистрации.

Идентификатор: `user_id`.

### Сущность `Категория`

Атрибуты:

- идентификатор категории;
- название категории;
- описание категории.

Идентификатор: `category_id`.

### Сущность `Рецепт`

Атрибуты:

- идентификатор рецепта;
- автор рецепта;
- категория рецепта;
- название;
- описание;
- время приготовления;
- количество порций;
- сложность;
- дата создания.

Идентификатор: `recipe_id`.

### Сущность `Ингредиент`

Атрибуты:

- идентификатор ингредиента;
- название ингредиента;
- единица измерения по умолчанию.

Идентификатор: `ingredient_id`.

### Сущность `Состав рецепта`

Эта сущность нужна для разрешения связи многие-ко-многим между рецептами и ингредиентами.

Атрибуты:

- идентификатор рецепта;
- идентификатор ингредиента;
- количество;
- единица измерения;
- примечание.

Идентификатор: составной ключ `(recipe_id, ingredient_id)`.

### Сущность `Шаг приготовления`

Атрибуты:

- идентификатор рецепта;
- номер шага;
- инструкция.

Идентификатор: составной ключ `(recipe_id, step_number)`.

### Сущность `Отзыв`

Атрибуты:

- идентификатор отзыва;
- идентификатор рецепта;
- идентификатор пользователя;
- оценка;
- комментарий;
- дата создания.

Идентификатор: `review_id`. Дополнительно задана уникальность пары `(recipe_id, user_id)`.

### Сущность `Избранное`

Атрибуты:

- идентификатор пользователя;
- идентификатор рецепта;
- дата добавления.

Идентификатор: составной ключ `(user_id, recipe_id)`.

## 3. Реляционная модель и НФБК

В реляционной модели все сущности преобразованы в таблицы. Связь многие-ко-многим между `recipe` и `ingredient` разрешена с помощью таблицы `recipe_ingredient`. Связь многие-ко-многим между `app_user` и `recipe` для избранного разрешена через таблицу `favorite`.

Основные типы данных выбраны в соответствии с хранимыми данными:

- идентификаторы хранятся в числовом типе `SERIAL`/`INT AUTO_INCREMENT`;
- названия хранятся в `VARCHAR`, а не в больших текстовых объектах;
- время приготовления и количество порций хранятся в целочисленном типе;
- количество ингредиента хранится в `NUMERIC`/`DECIMAL`;
- дата создания хранится в `TIMESTAMP`.

Доказательство соответствия НФБК:

1. Таблица `app_user` находится в НФБК, потому что все неключевые атрибуты зависят от ключа `user_id`. Также `username` и `email` являются альтернативными ключами, так как имеют ограничение уникальности.
2. Таблица `category` находится в НФБК, потому что `name` уникально, а описание зависит только от идентификатора категории или альтернативного ключа `name`.
3. Таблица `recipe` находится в НФБК, потому что все сведения о рецепте зависят от ключа `recipe_id`. Уникальная пара `(author_id, title)` также может рассматриваться как альтернативный ключ.
4. Таблица `ingredient` находится в НФБК, потому что название ингредиента уникально, а единица измерения зависит от идентификатора ингредиента или названия.
5. Таблица `recipe_ingredient` находится в НФБК, потому что количество, единица измерения и примечание зависят от полного составного ключа `(recipe_id, ingredient_id)`.
6. Таблица `recipe_step` находится в НФБК, потому что инструкция зависит от полного составного ключа `(recipe_id, step_number)`.
7. Таблица `review` находится в НФБК, потому что оценка и комментарий зависят от ключа `review_id`, а пара `(recipe_id, user_id)` является альтернативным ключом.
8. Таблица `favorite` находится в НФБК, потому что дата добавления зависит от полного составного ключа `(user_id, recipe_id)`.

Таким образом, в таблицах отсутствуют частичные и транзитивные зависимости неключевых атрибутов от ключей, а все функциональные зависимости имеют детерминант, являющийся ключом или альтернативным ключом.

## 4. Создание базы данных

Для проекта подготовлены две версии базы данных:

1. PostgreSQL — папка `sql/postgresql`.
2. MySQL — папка `sql/mysql`.

Для каждой СУБД подготовлены файлы:

- `01_create_schema.sql` — создание таблиц и ограничений;
- `02_insert_test_data.sql` — тестовое заполнение;
- `03_constraint_checks.sql` — проверка ограничений.

В тестовых данных добавлено не менее пяти записей для основных таблиц и не менее пяти записей для таблиц-связей.

## 5. Работа с двумя СУБД

Java-приложение работает с двумя СУБД через JDBC. Данные подключения берутся из файлов настроек:

- `config/app-postgres.properties`;
- `config/app-mysql.properties`.

Для переключения СУБД не нужно менять код приложения. Достаточно запустить приложение с другим конфигурационным файлом.

## 6. Клиентское приложение

Клиентское приложение написано на Java. Для подключения к базе данных используется JDBC. Фреймворки и объектно-реляционные оболочки не используются.

Главные классы приложения:

- `App` — запуск приложения и главное меню;
- `DbConfig` — чтение файла настроек;
- `Db` — создание подключения к базе данных;
- `RecipeService` — выполнение операций с рецептами, отзывами и избранным.

Функции приложения:

1. Просмотр всех рецептов.
2. Просмотр подробной информации о рецепте.
3. Поиск рецепта по названию.
4. Добавление нового рецепта.
5. Добавление отзыва.
6. Добавление рецепта в избранное.
7. Просмотр справочников пользователей, категорий и ингредиентов.

## 7. Тестирование

План тестирования находится в файле `docs/testing_plan.md`.

Проверяются:

- создание структуры базы данных;
- добавление тестовых данных;
- работа первичных и внешних ключей;
- работа ограничений `CHECK`;
- работа уникальных ограничений;
- запуск приложения с PostgreSQL;
- запуск приложения с MySQL;
- основные пользовательские сценарии;
- работа проекта на двух операционных системах.
